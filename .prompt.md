# Refactor Message Paging to use SeqId

## Goal
Replace `timeStamp` based pagination with `seqId` based pagination in `getMessagesByChatCode` API.
This improves reliability and aligns with the new sequence-based architecture.

## Changes

### Backend
1.  **MessageMapper** (`.java` & `.xml`):
    *   Change `selectMessageListByChatIdAndTimeStamp` to `selectMessageListByChatIdAndCursor`.
    *   Param: `cursorSeqId` (Long) instead of `timeStamp`.
    *   SQL: `WHERE chat_id = #{chatId}`. If `cursorSeqId != null`, add `AND seq_id < #{cursorSeqId}`. `ORDER BY seq_id DESC LIMIT 50`.
2.  **MessageService** (`.java` & `Impl.java`):
    *   Update `getMessagesByChatCode` signature: `(Integer userId, String chatCode, Long cursorSeqId)`.
    *   Pass `cursorSeqId` to mapper.
3.  **MessageController**:
    *   Update endpoint `@GetMapping`: `@RequestParam(required = false) Long cursorSeqId`.

### Frontend
1.  **Message.ts** (`src/api/Message.ts`):
    *   Update `getMessageListApi` request param from `createTime` to `cursorSeqId`.
    *   Update URL construction.
2.  **messageStore.ts** (`src/stores/messageStore.ts`):
    *   Update `getMessageList` to accept `cursorSeqId` (number).
    *   In "load more" call, pass `oldestMessage?.seqId`.

## Verification
-   **Compile**: Java backend.
-   **Review**: Ensure frontend calls match backend params.
